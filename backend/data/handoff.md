## Design Decisions Made This Session

### `work_auth` omitted from `NoteEntityType` (deviation from roadmap text)

The earlier roadmap prose listed `work_auth` as an entity type. Dropped it for now — closure gating doesn't need it and no concrete use case exists. Easy to add later (enum value + one more branch in `get_blocking_notes_for_project`). The roadmap was updated to reflect the actual scope.

### `get_blocking_notes_for_project()` lives in `app/projects/service.py`

Not in `app/notes/service.py`. Rationale: it's a project-scoped aggregation that has to join across multiple foreign modules (time_entries, deliverables, sample_batches, project itself). Keeping it in `projects/service.py` avoids pulling all those models into `notes/`. `app/notes/service.py` owns note CRUD, system-note primitives, and resolution logic only.

### Self-referential `Note.parent` / `Note.replies` use `remote_side="Note.id"`

The "remote side" in a self-ref relationship is the **parent** side (the one that holds the target of the FK). Local FK is `parent_note_id`, pointing to `id` on the parent row, so `remote_side="Note.id"` on the `parent` relationship is correct. Replies use `back_populates="parent"` with `cascade="all, delete-orphan"`.

### Full Pydantic schema set in Session A (not just `NoteRead`)

All four schemas (`Create`, `Reply`, `Resolve`, `Read`) landed in Session A even though endpoints don't exist yet. Keeps Session A self-contained — Session C just plugs them into routers.

### Placeholder `app/common/models.py:Note` deleted outright

Rather than superseding it. Never migrated, no importers anywhere in the codebase; removing it is the cleanest outcome. The stub used `parent_type`/`parent_id`; the new design uses `entity_type`/`entity_id` (matches service function signatures like `create_system_note(entity_type, entity_id, ...)`).

## Non-Obvious Technical Patterns (carried forward)

### Self-referential relationship with `remote_side`

For a one-to-many self-ref (parent → children) where the FK lives on the child, set `remote_side="Model.id"` on the `parent` relationship, and pair it with a reciprocal `back_populates`-linked collection on the parent. Getting `remote_side` pointed at the wrong column yields a silently-wrong relationship that loads nothing or loops.

### Recursive Pydantic schemas need `model_rebuild()`

`NoteRead` references itself via `replies: list["NoteRead"] = []`. Pydantic v2 requires an explicit `NoteRead.model_rebuild()` after the class definition for the self-reference to resolve.

### Polymorphic attachment — no DB-level FK

`notes.entity_id` has no FK. Service layer validates entity existence before insert. Deleting the parent entity does not cascade to its notes — that is explicitly app-layer (or deferred) cleanup.

### Existing patterns still in force

See `data/handoff.md` history and `app/PATTERNS.md`: `db.get()` vs `select() + populate_existing`, FK validation in early-return paths, `PermissionChecker` returns the user, audit-field test pattern, user-managed migrations, `lazy="selectin"` on serialized relationships, SQLite `Numeric` cast via `Decimal(str(value))`.
